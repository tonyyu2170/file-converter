# Phase 21 — COOP/COEP + multi-threaded ffmpeg (stub for parallel-instance handoff)

**Date:** 2026-05-06
**Status:** stub — written for a parallel Claude instance to brainstorm and plan from
**Source of truth:** `docs/superpowers/specs/2026-05-05-v2-design.md` §2.2 (deployment commitments), §11 (phasing item 4)

> **Read this first if you are the parallel instance:** this is not a finished spec. It is a one-page contract that pins what Phase 21 must do and must not do, plus open design questions you should brainstorm with the user before writing the implementation plan. The full design context is in the v2 design doc cited above. Phase 20 is being implemented in parallel on branch `phase-20-trim-scrubber-and-audio-trim`; Phase 21 lands on top after rebase.

## 1. Goal

Enable multi-threaded ffmpeg.wasm by:

1. Adding `Cross-Origin-Opener-Policy: same-origin` and `Cross-Origin-Embedder-Policy: require-corp` response headers to the static site.
2. Swapping `@ffmpeg/core` (single-threaded) for `@ffmpeg/core-mt` (multi-threaded) in `package.json`, the core-copy pipeline, and `loadFfmpeg()` URL pinning.
3. Auditing the entire app for cross-origin embeds that `require-corp` would break, since v1 was audited 2026-05-05 but every PR that touches layout, fonts, or third-party assets must re-verify (per v2 design §2.2).

This unlocks ~2× to ~4× speedups for ffmpeg operations once `video-convert` lands in Phase 22.

## 2. Scope

### In

- `vercel.json`: add the two COOP/COEP headers to the existing `headers()` rule that currently sets `Cross-Origin-Resource-Policy` and CSP.
- `package.json`: replace `@ffmpeg/core` dependency with `@ffmpeg/core-mt` of the same minor version.
- `pnpm-lock.yaml`: regenerated by `pnpm install`.
- `scripts/copy-ffmpeg-core.mjs`: source path swaps from `node_modules/@ffmpeg/core/dist/umd/*` to `node_modules/@ffmpeg/core-mt/dist/umd/*`. The MT build adds a worker file (`ffmpeg-core.worker.js`); the script must copy it too.
- `scripts/ffmpeg-manifest.json`: pinned versions and sha256s for the new MT files.
- `src/engines/_shared/ffmpeg/index.ts`: the `coreURL` and `wasmURL` pin same-origin paths; add the `workerURL` pin for the MT worker. **Public API (`loadFfmpeg()` signature and return shape) does not change.**
- COEP audit: walk every page (`/`, every `/tools/<id>`, `/about`, `/privacy`) and confirm no cross-origin `<img>`, `<iframe>`, font, script, or stylesheet that lacks `Cross-Origin-Resource-Policy: cross-origin` headers from the source. Document what was audited.

### Out

- New engines (Phase 22+).
- Any change to `src/engines/_shared/trim-scrubber/`, `src/engines/audio-trim/`, `src/engines/audio-convert/`, the home grid, or the sidebar (these are Phase 20's surface).
- Any CSP changes (`'wasm-unsafe-eval'` in `script-src` already covers the MT build per v2 §2.2).

## 3. Phase 20 coordination contract

Phase 20 (parallel branch `phase-20-trim-scrubber-and-audio-trim`) promises:

1. No edits to `vercel.json`, `package.json`, `pnpm-lock.yaml`, `scripts/copy-ffmpeg-core.mjs`, `scripts/ffmpeg-manifest.json`, `src/engines/_shared/ffmpeg/index.ts`, `next.config.ts`.
2. Calls `loadFfmpeg()` from `_shared/ffmpeg/` only, never imports `@ffmpeg/core` or `@ffmpeg/core-mt` directly.

Phase 21 must promise:

1. No edits to `src/engines/_shared/trim-scrubber/`, `src/engines/audio-trim/`, `src/engines/audio-convert/`, the home grid, or the sidebar.
2. **`loadFfmpeg()` public surface unchanged.** Same parameters, same return type. The MT switch is internal: under the hood, `loadFfmpeg()` now resolves to a `@ffmpeg/core-mt` instance; consumers don't notice. Phase 20 audio-trim / Phase 19 audio-convert keep working without code changes.
3. Phase 19's audio-convert and Phase 20's audio-trim correctness E2E tests still pass after the swap. (Single-threaded → multi-threaded transcode produces bit-identical output for `-c copy` and equivalent quality for re-encodes; the `(±50 ms duration, RIFF/WAVE magic bytes)` style assertions in Phase 19/20 should still hold.)

Merge order: Phase 20 first → Phase 21 rebases.

## 4. Open questions for brainstorming

These are unresolved and the parallel instance should drive them through the brainstorming flow before writing the implementation plan.

### 4.1 COEP audit findings — what does the app actually embed?

v1 design says "v1 doesn't embed any (audited 2026-05-05)". Re-verify after Phases 14–20 landed: walk the home page, every tool page, the about page, the privacy page. Common breakers to look for:

- Google Fonts (often served `Cross-Origin-Resource-Policy: cross-origin` on the font CSS but check the font files too).
- External images (favicons, logos, demo screenshots).
- Third-party scripts (analytics, error reporters — should be none for this project but verify).
- `<iframe>` embeds.

If anything embeds cross-origin without `Cross-Origin-Resource-Policy: cross-origin`, options are: (a) self-host, (b) serve via a proxy that adds CORP, (c) drop the embed. Each PR that touches assets in v2 must repeat this check.

### 4.2 MT worker file pinning

`@ffmpeg/core-mt` ships `ffmpeg-core.worker.js` in addition to the JS+WASM pair. The hash check in `scripts/copy-ffmpeg-core.mjs` must extend to cover it. Decide whether to keep one manifest entry per file (cleaner) or one entry containing all three sha256s (more compact).

### 4.3 Local dev under COOP/COEP

`pnpm dev` runs the Next.js dev server which doesn't honor `vercel.json` headers. To exercise multi-threaded ffmpeg locally you need COOP/COEP at the dev server too. Options:

- Add a `next.config.ts` `headers()` rule (works in dev but no-op in production static export).
- Add a small middleware-style Node script that wraps `next dev` and injects headers.
- Document the `pnpm build && npx serve out -L --headers ...` workflow as the only way to test MT locally.

Pick one. Document it in `CLAUDE.md` if it changes the daily workflow.

### 4.4 Browser support fallback

Some users browse with `SharedArrayBuffer` disabled (older Safari, certain corporate proxies). The MT build refuses to load without `crossOriginIsolated === true`. Options:

- Hard fail with an actionable error (matches the privacy-first stance).
- Detect at boot and dynamically import `@ffmpeg/core` (single-threaded) instead, accepting the cost of shipping both cores.
- Detect at boot and refuse, but show a guide page.

This is a UX decision and should be brainstormed with the user.

### 4.5 Test gating

Multi-threaded ffmpeg performance is best validated against video transcodes which don't ship until Phase 22. For Phase 21, the regression check is "audio-convert and audio-trim still work after the swap." Decide whether to expand `RUN_AUDIO_CONVERT_CORRECTNESS=1` / `RUN_AUDIO_TRIM_CORRECTNESS=1` to include a transcode-time check, or accept that Phase 22 will validate the speedup.
