# v2 — design

**Status:** approved 2026-05-05
**Owner:** Tony Yu
**Predecessors:** master spec `2026-04-30-file-converter-design.md`; v1 closeout (`57f4b23 chore(release): bump version to v1.0.0`)

## 0. Goal

v2 is a single cohesive release that does five things at once:

1. Introduces **audio** conversion + trimming via ffmpeg.wasm.
2. Introduces **video** conversion, trimming, and audio extraction via the same ffmpeg.wasm (gated behind COOP/COEP).
3. Introduces **OCR** (image → text) via Tesseract.js.
4. Adds **archive** create/extract (ZIP / TAR / TAR.GZ).
5. Adds **structured-data** conversion (CSV / JSON / YAML / XML).

In addition, v2 closes the one known v1 quality gap by replacing the portrait-only `image-bg-remove` model with a general-purpose alternative.

The catalog grows from 13 → 24 engines. The architectural commitments are real: a new shared ffmpeg.wasm pipeline, a new shared Tesseract.js pipeline, a new shared trim-scrubber UI primitive, and the COOP/COEP headers required for `SharedArrayBuffer` in the browser.

The defining property — **files never leave the browser** — is preserved. CSP stays at `connect-src 'self'`; `scripts/check-bundle-isolation.mjs` extends to verify the new shared infra is lazy-loaded; the privacy E2E continues to assert zero outbound network during conversion across all 24 engines.

## 2. Architectural commitments

### 2.1 New shared infrastructure

#### `src/engines/_shared/ffmpeg/`

Single ffmpeg.wasm load, shared by `audio-convert`, `audio-trim`, `video-convert`, `video-trim`, `video-extract-audio`. Lazy-loaded; first use of any of these engines pays the ~30 MB cost once and the result is cached for the page-life. Subsequent engines reuse the same instance.

Public surface:

```ts
export async function loadFfmpeg(): Promise<FFmpeg>;     // cached singleton
export type FFmpegProgress = { percent: number; phase?: string };
```

Workers import lazily via `await import("@/engines/_shared/ffmpeg")` to keep the import out of the homepage chunk.

#### `src/engines/_shared/tesseract/`

Tesseract.js core + English language pack. Same lazy-load + cache model as ffmpeg. Used only by `image-to-text` in v2; structured to accept additional language packs in v2.1.

Public surface:

```ts
export async function loadTesseract(lang: "eng"): Promise<Worker>;
```

#### `src/engines/_shared/trim-scrubber/`

New shared UI primitive used by `audio-trim` and `video-trim`. Lives in `_shared/` (not `src/components/`) because it has engine-specific knowledge (waveform decode, frame-strip extraction).

Component shape:

```ts
type TrimScrubberProps = {
  source: File;
  modality: "audio" | "video";
  durationSec: number;
  startSec: number;
  endSec: number;
  onChange(start: number, end: number): void;
  disabled?: boolean;
};
```

- **Audio mode** decodes via `OfflineAudioContext`, downsamples PCM to ~512 samples, renders as a vertical-bars canvas.
- **Video mode** asks ffmpeg to seek-and-output 8 PNG thumbnails at evenly-spaced timestamps, renders as a horizontal strip.
- Two drag handles for start/end. Keyboard support (left/right arrows; shift = jump 10× larger). Read-only `mm:ss.ms` labels under each handle.
- Same brutalist treatment as `pdf-edit` page tray: monospace timestamp labels, sharp hairline handles, no gradients.

### 2.2 Deployment commitments (`vercel.json`)

- **`Cross-Origin-Opener-Policy: same-origin`** + **`Cross-Origin-Embedder-Policy: require-corp`** — required for `SharedArrayBuffer`, which ffmpeg.wasm needs for video transcode performance.
- Long-lived cache headers on `.wasm` and `.onnx` (already in v1; extends to new files via wildcard).
- CSP unchanged. `'wasm-unsafe-eval'` already in `script-src` covers both ffmpeg and Tesseract. `connect-src 'self'` still holds.

`require-corp` can break embedded cross-origin resources. v1 doesn't embed any (audited 2026-05-05); every PR in v2 that touches layout, fonts, or third-party assets re-verifies.

### 2.3 Engine pattern unchanged

All 11 new engines plug into the existing `ConversionEngine` interface (master spec §6.3). UI components, hooks, and routes do not change for v2 — `_shared/harness.ts` adapts the UI generically based on engine metadata, exactly as in v1. Adding each engine is a single folder under `src/engines/<id>/` plus one registry line in `src/engines/_shared/registry.ts`.

The `EngineCategory` type in `src/engines/_shared/types.ts:33` extends from `"image" | "pdf" | "document"` to add `"audio" | "video" | "archive" | "data" | "ocr"`.

## 3. Per-family engine specs

### 3.1 Audio

#### `audio-convert`

- **Input:** single file; `.mp3`, `.wav`, `.m4a`, `.flac`.
- **Output:** single audio file in selected format.
- **Options:**
  - `outputFormat: "mp3" | "wav" | "m4a" | "flac"`.
  - `bitrate: 64 | 128 | 192 | 256 | 320` (kbps; default 192). UI hides this field for lossless outputs (wav / flac).
- **Worker:** ffmpeg `-i in -c:a {codec} -b:a {br}k out`.
- **Validation:** extension + magic bytes; size cap 500 MB.

#### `audio-trim`

- **Input:** single audio file (same formats as `audio-convert`).
- **Output:** single trimmed audio file.
- **Options:**
  - `startSec: number`, `endSec: number`.
  - `outputFormat: "same" | "mp3" | "wav" | "m4a" | "flac"` (default `"same"`).
- **Worker:** ffmpeg `-i in -ss S -to E -c copy out` when format unchanged (near-instant, no quality loss); re-encode pipeline when format changes.
- **UX:** uses `_shared/trim-scrubber` with `modality: "audio"`. Default output `"same"` so most users get a fast lossless trim.

### 3.2 Video

#### `video-convert`

- **Input:** single file; `.mp4`, `.mov`, `.webm`. Size cap **100 MB** (constrained by ffmpeg.wasm memory + transcode time).
- **Output:** single video file in selected format.
- **Options:**
  - `outputFormat: "mp4" | "mov" | "webm"`.
  - `quality: "low" | "medium" | "high"` mapping to CRF 28 / 23 / 18 with sane preset.
- **Worker:** ffmpeg full transcode.
- **UX:** mandatory progress UI with phase + percent; **Convert** button transforms into **Cancel** which calls `signal.abort()` → ffmpeg terminate. Pre-conversion tooltip: "this typically takes ~1 minute per minute of video."

#### `video-trim`

- **Input:** single video file (same formats); 100 MB cap.
- **Output:** single trimmed video file.
- **Options:** `startSec: number`, `endSec: number`.
- **Worker:** ffmpeg `-i in -ss S -to E -c copy out` (no re-encode).
- **UX:** uses `_shared/trim-scrubber` with `modality: "video"`. Tooltip: "cuts may snap to nearest keyframe — typically within 1 second."

#### `video-extract-audio`

- **Input:** single video file (same formats); 100 MB cap.
- **Output:** single audio file.
- **Options:**
  - `outputFormat: "mp3" | "m4a" | "wav"`.
  - `bitrate: 64 | 128 | 192 | 256 | 320` (lossy formats only; default 192).
- **Worker:** ffmpeg `-i in -vn -c:a {codec} -b:a {br}k out`.
- **UX:** reuses `audio-convert`'s bitrate dropdown component.

### 3.3 Archives

#### `archive-extract`

- **Input:** single archive file. Auto-detected from magic bytes: ZIP, TAR, TAR.GZ, TGZ. 200 MB cap.
- **Output:** **multi** — one `OutputItem` per entry. ResultList packages output as a download bundle (existing pattern; cf. `pdf-split`). Engine descriptor sets `archiveSuffix: "-extract"`.
- **Options:** none.
- **Worker:** fflate (zip) + pako (gzip) + tar parser.
- **Validation:**
  - Encrypted ZIPs rejected with actionable error: "this archive is encrypted; password-protected ZIPs aren't supported."
  - Entry paths sanitized against zip-slip path traversal. Any entry containing `../` or an absolute path **aborts the entire extraction** with an actionable error ("archive contains entry with unsafe path: `<path>`; refusing to extract"). No partial extraction — privacy and security default to strict.
  - Per-entry size sanity check before extracting (reject single-entry archives that would expand to > 1 GB).

#### `archive-create`

- **Input:** **multi** via StagingArea (same UX as `pdf-merge`). Sum 500 MB across all inputs.
- **Output:** single archive file.
- **Options:**
  - `outputFormat: "zip" | "tar.gz"` (default `"zip"`).
  - `filename: string` (default `archive-YYYYMMDD-HHmm`, extension appended).
- **Worker:** client-zip for ZIP; pako + tar for TAR.GZ.
- **UX:** order in StagingArea preserved in archive.

### 3.4 Data

#### `data-convert`

- **Input:** single file; `.csv`, `.json`, `.yaml`, `.yml`. Format auto-detected via extension + sniff. 50 MB cap.
- **Output:** single text file in selected format.
- **Options:** `outputFormat: "csv" | "json" | "yaml"`.
- **Worker:** papaparse (CSV) + js-yaml (YAML) + native JSON. JS object as intermediate.
- **Validation:**
  - CSV → tree assumes header row.
  - Tree → CSV requires shallow-tabular shape. Deeply nested input rejected with actionable error: "row 3 contains a nested object — flatten or pick JSON / YAML output."

#### `json-format`

- **Input:** single `.json` file. 50 MB cap.
- **Output:** single `.json` file with formatting applied.
- **Options:**
  - `mode: "pretty" | "minify"` (default `"pretty"`).
  - `indent: 2 | 4 | "tab"` (pretty mode only; default `2`).
- **Worker:** `JSON.parse` + `JSON.stringify`.

#### `xml-to-json`

- **Input:** single `.xml` file. 50 MB cap.
- **Output:** single `.json` file.
- **Options:**
  - `attributePrefix: "@" | "$_" | ""` (default `"@"`, Badgerfish-inspired).
- **Worker:** fast-xml-parser.
- **Notes:** one-way conversion. JSON → XML reconstruction is lossy and not in v2.

### 3.5 OCR

#### `image-to-text`

- **Input:** single image file; `.jpg`, `.jpeg`, `.png`, `.webp`, `.heic` (HEIC reuses libheif from `image-convert` via `_shared/`). 25 MB cap.
- **Output:** single `.txt` file (default) or `.json` with bounding boxes.
- **Options:** `outputFormat: "txt" | "json-with-bboxes"` (default `"txt"`).
- **Worker:** `_shared/tesseract` with English language pack.
- **UX:** progress UI driven by Tesseract.js's progress callback (mandatory — recognition takes seconds-to-minutes). Pre-conversion tooltip: "best on scanned documents and screenshots; lower quality on photos."

### 3.6 Quality fix: `image-bg-remove` model swap

Same engine boundary, new weights + threshold tuning.

- **Lead candidate:** **BiRefNet-lite int8** (~40 MB, general-purpose, Apache 2.0). MODNet was ~25 MB; net add ≈ 15 MB.
- **Hard prerequisite (Phase 18 verification gate):** OOM verification on 8 GB dev box across 1080p portrait, 1080p landscape non-portrait, 2K landscape, 4K landscape. Document the achievable cap. If the lead candidate OOMs at 2K, fall back to ISNet-DIS-tiny or U2Net quantized; final selection is recorded in the Phase 18 plan.
- **Test fixtures:** add committed non-portrait fixtures (object on plain bg, animal, indoor scene) with expected-mask snapshots. Existing portrait fixture kept to verify regression-free behavior.
- **/about engines table:** the "portrait-optimized" footnote on this row is removed in Phase 26.

## 4. UX surfaces

### 4.1 Sidebar (`src/components/layout/sidebar.tsx`)

- **`EngineCategory` extension** drives new sidebar groups: `AUDIO`, `VIDEO`, `ARCHIVES`, `DATA`, `OCR`.
- **`GROUP_ORDER` becomes:** `HOME → IMAGES → PDFS → DOCS → AUDIO → VIDEO → ARCHIVES → DATA → OCR → ABOUT`.
- **Sidebar density:** at 24 engines + 9 group headings (~33 lines), the 180 px column needs `overflow-y: auto`. No collapsing — the brutalist aesthetic is "everything visible." Horizontal scroll never; sidebar width unchanged.

### 4.2 Home page grid (`src/app/page.tsx`)

Currently a flat grid of all 13 engine cards. v2 sections the grid by category — five new section blocks rendered in the same order as the sidebar. Each section gets a `[ section ]`-style header consistent with the rest of the site. ~50 lines of section-header markup + a `groupBy(engine, e => e.category)` call. Card markup itself unchanged.

### 4.3 Per-engine routes

Each new engine gets `src/app/tools/<id>/page.tsx` following the existing `ToolFrame` pattern. No shared-code changes. Each engine ships its own `OptionsPanel` in `src/engines/<id>/options.tsx`.

### 4.4 `/about` engines table

Auto-rendered from registry (Phase 17 infrastructure). Grows from 13 rows → 24 rows. Each new engine ships with `library` and `license` populated on its `EngineMeta`. The bg-remove row's "portrait-optimized" footnote is removed in Phase 26.

### 4.5 Slow-engine progress + cancel

ffmpeg.wasm video transcode + Tesseract.js OCR + bg-remove inference are all noticeably slow. v1's `ConversionProgress` type already supports phased progress; v2 ensures every slow engine reports phase + percent + ETA, and the **Convert** button transforms into a **Cancel** button that calls `signal.abort()`. The `signal: AbortSignal` is already in the engine contract; v2 verifies it's wired through to ffmpeg / tesseract / transformers.js cancellation.

## 5. Phasing strategy

v2 internally splits into 9 sequential phases. Dependency + risk distribution shaped this order:

| # | Phase | Engines / scope | New infra | Risk |
|---|---|---|---|---|
| 18 | bg-remove model swap | (existing engine, new weights) | OOM verification on dev box | Low. Independent; could ship as v1.1 if v2 stalls. |
| 19 | ffmpeg infra + audio-convert | `audio-convert` | `_shared/ffmpeg/` | Medium. New shared WASM pattern validated on the simpler audio path before COOP/COEP. |
| 20 | trim-scrubber + audio-trim | `audio-trim` | `_shared/trim-scrubber/` | Medium. New shared UI primitive; audio half tests waveform render. |
| 21 | COOP/COEP + video-convert | `video-convert` | `vercel.json` headers; SAB enablement | **High.** Single riskiest phase. Concentrated here so a regression doesn't pollute later phases. |
| 22 | video-trim + video-extract-audio | `video-trim`, `video-extract-audio` | (none new) | Medium. Trim-scrubber's video render path tested here. |
| 23 | tesseract infra + image-to-text | `image-to-text` | `_shared/tesseract/` | Medium. Independent of ffmpeg track. |
| 24 | archives | `archive-extract`, `archive-create` | (none new) | Low. JS-lib glue. |
| 25 | data | `data-convert`, `json-format`, `xml-to-json` | (none new) | Low. All three small. |
| 26 | v2 closeout | sidebar groups, home-grid sectioning, /about table refresh, vercel.json final review, a11y sweep, Lighthouse, securityheaders | (none new) | Low. Cross-cutting QA. Mirrors Phase 17. |

**Per-phase invariant:** lint + typecheck + unit + integration + privacy E2E + bundle-isolation gate all green. No phase merges with regressions. `main` is always shippable.

**Parallelization opportunity.** Phase 23 is independent of the entire ffmpeg track (Phases 19–22). If `superpowers:subagent-driven-development` is the execution mode, Phase 23 can land in parallel with one of 19–22. Phases 24–25 are also fully independent and can both run in parallel with the ffmpeg track once Phase 18 is done. Sequential execution remains the safe default.

**v1.1 escape hatch.** If v2 stalls at any point, Phase 18 alone is a meaningful patch release that closes the one known v1 quality gap. Phase 18 is designed first specifically to preserve this option.

### 7.3 Bundle and cache

- ffmpeg.wasm core (~30 MB) loaded on first use of any audio/video engine; cached at module level for the page-life. Subsequent engines reuse the same instance.
- Tesseract core (~10 MB) + English lang pack (~10 MB) loaded on first navigation to `/tools/image-to-text`.
- BiRefNet-lite int8 (~40 MB) loaded on first navigation to `/tools/image-bg-remove`. `scripts/copy-bg-models.mjs` and `scripts/fetch-bg-models.mjs` extend to handle the new model.
- HTTP caching: existing `Cache-Control: public, max-age=31536000, immutable` on `.wasm` / `.onnx` extends to all new files via wildcard in `vercel.json`.
- CSP unchanged.
- First-use loading state on slow engines reads "Loading audio engine (≈30 MB; cached after first use)" so the wait is contextualized.

## 8. Risks and open questions

### 8.1 Risk register (in priority order)

1. **COOP/COEP regression breaking embedded resources.** Phase 21 verifies all existing routes still work after header rollout. If any cross-origin asset breaks, fix it (CORP header on the source, or self-host) before merging Phase 21.
2. **ffmpeg.wasm video transcode being slower than users tolerate.** Tooltip sets expectation pre-conversion; progress UI is honest; cancel works. Acceptance is "users understand the wait," not "wait is short."
3. **bg-remove model OOMing on 8 GB dev box at 4K input.** Phase 18 verification gate across 4 fixture sizes; size cap and pre-conversion warning if 4K isn't reachable; fallback model chain documented.
4. **Bundle weight scaring users on first audio/video conversion.** Lazy-load is hard-enforced by bundle-isolation script; first-use loading state explains the size and that it's a one-time download.
5. **Tesseract OCR quality being underwhelming on photos.** Tooltip sets expectation ("best on scanned documents and screenshots").
6. **`-c copy` trim cuts snapping to keyframes producing slight inaccuracies.** Documented in tooltip. Frame-accurate cuts (re-encode mode) are explicitly v2.1 territory.

### 8.2 Open questions

- **bg-remove model selection.** BiRefNet-lite int8 is the lead candidate; final choice deferred to Phase 18 OOM verification data. The Phase 18 plan records the actual selection.
- **Sidebar overflow on short viewports.** v2 closeout adds `overflow-y: auto`. On a 600 px-tall window, the sidebar will scroll independently of the main column. Acceptable.
